# BizGuard v1.2.2 — Fix: OAuthCallbackError en SSO con Microsoft Entra ID

**Fecha:** 2026-03-21
**Versión:** v1.2.2-Enterprise
**Estado:** ✅ Resuelto y en producción

---

## Contexto del sistema

BizGuard es un reverse proxy enterprise construido sobre Next.js 16.2 (App Router, standalone build). Expone múltiples conectores, cada uno con su propio `publicHost` y puerto:

| Conector | Puerto | Host Público | Target |
|---|---|---|---|
| `core-serena` | 8080 | `core.nathium.com.ar` | `https://arbuevcsa.bizland.tech/` |
| `test-port` | 8081 | `core2.nathium.com.ar` | `https://awx.bizland.tech/` |

El dashboard de administración corre en `localhost:3000` con **bypass de autenticación** — el SSO protege únicamente el acceso a los conectores.

**Flujo de tráfico:**
```
Internet → Cloudflare Tunnel → BizGuard proxy (puerto 8080/8081)
         → internal-dashboard proxy → localhost:3000 (Next.js)
```

---

## Síntoma

Al intentar acceder a `https://core.nathium.com.ar/`, el flujo SSO con Microsoft Entra ID fallaba en loop con:

```json
{
  "name": "q",
  "type": "OAuthCallbackError",
  "kind": "signIn"
}
```

El error se repetía indefinidamente — cada intento fallaba con el mismo error sin causa visible.

---

## Análisis del flujo OAuth/PKCE

### Flujo correcto esperado

```
1. Usuario → https://core.nathium.com.ar/
2. proxy-manager → 302 /api/auth/signin?callbackUrl=https://core.nathium.com.ar/
3. NextAuth → 302 /login?callbackUrl=...  (pages.signIn = '/login')
4. login page carga → useEffect → signIn("microsoft-entra-id", { callbackUrl })
5. POST /api/auth/signin/microsoft-entra-id → CREATE_PKCE + authorization URL
6. Browser → https://login.microsoftonline.com/.../authorize?...&redirect_uri=X&code_challenge=Y
7. Microsoft autentica → 302 https://core.nathium.com.ar/api/auth/callback/microsoft-entra-id?code=Z&state=W
8. NextAuth callback → USE_PKCE → PARSE_PKCE → token exchange → sesión creada
```

### Lo que mostraban los logs

```
[Auth TRACE] CREATE_PKCECODEVERIFIER {"payload":"tX9Due1M..."}
[Auth TRACE] authorization url is ready {"url":"...redirect_uri=https%3A%2F%2Fcore.nathium.com.ar%2Fapi%2Fauth%2Fcallback%2Fmicrosoft-entra-id..."}
[Auth TRACE] USE_PKCECODEVERIFIER  ← verifier leído del cookie (correcto)
[Auth TRACE] PARSE_PKCECODEVERIFIER ← mismo valor que USE (sin corrupción)
[Auth TRACE] CLEAR_PKCECODEVERIFIER
[Auth ERROR DETAIL] {"type": "OAuthCallbackError"}
```

**Observaciones clave:**
- El PKCE verifier era leído correctamente (USE = PARSE = mismo valor)
- No había overwrite del cookie PKCE entre intentos
- La `authorization url` tenía el `redirect_uri` correcto: `https://core.nathium.com.ar/...`
- El error ocurría **después** de CLEAR_PKCE, es decir, durante el **token exchange** con Microsoft
- No había `error.cause` en el error — Microsoft rechazaba la solicitud antes de dar detalles

## Causa raíz #1: PKCE race condition (identificada y corregida antes)

**Descripción:** La página de login renderea `<StatsProvider>` (via `layout.tsx`) que crea un `EventSource("/api/stats/stream")`. Al resolver esta URL relativa en el contexto del conector (`https://core.nathium.com.ar/api/stats/stream`), el proxy-manager no encontraba sesión y lanzaba un **segundo redirect 302 a `/api/auth/signin`**, creando un **segundo flujo PKCE paralelo** que sobreescribía el cookie `authjs.pkce.code_verifier`.

**Consecuencia:** Cuando Microsoft respondía con el código para el primer flujo (challenge A), NextAuth leía el verifier del segundo flujo (verifier B) → mismatch → OAuthCallbackError.

**Fix aplicado en `proxy-manager.ts`:**

```typescript
if (!hasSession) {
  // Rutas /api/ NO deben iniciar flujo SSO — retornan 401.
  // Previene múltiples flujos PKCE concurrentes que corrompen los cookies de auth.
  if (url.startsWith('/api/') && !url.startsWith('/api/auth')) {
    this.log(`[BIZGUARD-Auth] API route ${url} requires auth. Returning 401.`, "info");
    res.writeHead(401, { 'Content-Type': 'application/json' });
    res.end(JSON.stringify({ error: 'Unauthorized' }));
    return;
  }
  // redirect normal a login para páginas navegables
  ...
}
```

Este fix fue **necesario pero insuficiente** — el error persistía porque había una segunda causa raíz.

## Causa raíz #2: Mismatch de `redirect_uri` en token exchange (causa principal del OAuthCallbackError)

### Análisis técnico profundo

En el flujo OAuth con PKCE, Azure AD valida que el `redirect_uri` en el **token exchange** (POST al token endpoint de Microsoft) sea **exactamente igual** al `redirect_uri` enviado en la **authorization URL**.

#### ¿Cómo NextAuth v5 computa el `redirect_uri`?

Trazando el código fuente de `@auth/core`:

**Para la Authorization URL** (`lib/actions/signin/authorization-url.js`):
```javascript
let redirect_uri = provider.callbackUrl;
if (!options.isOnRedirectProxy && provider.redirectProxyUrl) {
    redirect_uri = provider.redirectProxyUrl;  // usa redirectProxyUrl si está seteado
}
const params = Object.assign({
    redirect_uri,
    ...provider.authorization?.params,  // override si está seteado en el provider
}, ...)
```

**Para el Token Exchange** (`lib/actions/callback/oauth/callback.js`):
```javascript
let redirect_uri = provider.callbackUrl;  // ← usa callbackUrl, NO authorization.params
if (!options.isOnRedirectProxy && provider.redirectProxyUrl) {
    redirect_uri = provider.redirectProxyUrl;
}
let codeGrantResponse = await o.authorizationCodeGrantRequest(..., redirect_uri, ...);
```

**`provider.callbackUrl` se computa en** `lib/utils/providers.js`:
```javascript
const url = new URL(config.basePath ?? "/auth", params.url.origin);
callbackUrl: `${url}/callback/${id}`,
// params.url viene de internalRequest.url = new URL(req.url)
// req.url es la URL del request al servidor Next.js standalone
```

#### El problema

Next.js standalone server escucha en `0.0.0.0:3000` (HTTP). Cuando el browser hace la petición de callback a `https://core.nathium.com.ar/api/auth/callback/microsoft-entra-id`:

1. Cloudflare → puerto 8080 (BizGuard proxy)
2. proxy-server → reenvía a `http://127.0.0.1:3000` con headers:
   - `host: core.nathium.com.ar`
   - `x-forwarded-proto: https`
   - `x-forwarded-host: core.nathium.com.ar`

3. Next.js construye `Request.url` usando la **dirección de binding del servidor** (`0.0.0.0:3000`) y el protocolo del header forwarded:
   - `req.url = "https://0.0.0.0:3000/api/auth/callback/microsoft-entra-id"`

4. `params.url.origin = "https://0.0.0.0:3000"`

5. `provider.callbackUrl = "https://0.0.0.0:3000/api/auth/callback/microsoft-entra-id"`

**Resultado:**

| | `redirect_uri` |
|---|---|
| **Authorization URL** (nuestra config) | `https://core.nathium.com.ar/api/auth/callback/microsoft-entra-id` ✅ |
| **Token Exchange** (calculado internamente) | `https://0.0.0.0:3000/api/auth/callback/microsoft-entra-id` ❌ |

Azure AD requiere que ambos sean **idénticos** → rechaza → OAuthCallbackError sin causa detallada.

**Por qué la authorization URL era correcta:** Teníamos `authorization.params.redirect_uri = dynamicCallbackUrl` que sobreescribía el redirect_uri en el paso de autorización. Pero este override **no afecta el token exchange** — ese usa `provider.callbackUrl` directamente.

**Por qué `trustHost: true` no lo resolvía:** `trustHost: true` afecta la validación de cookies y la confianza de headers forwarded, pero no cambia cómo Next.js standalone construye `req.url` (que usa el binding address del servidor).

**Por qué `AUTH_TRUST_HOST=true` tampoco:** El env var activa `trustHost` pero no puede cambiar el comportamiento de construcción de URL del servidor standalone.

---

## Solución implementada

### `redirectProxyUrl` — el mecanismo correcto de auth.js

Auth.js tiene soporte para `redirectProxyUrl` diseñado exactamente para este patrón: cuando el servidor está detrás de un proxy y la URL pública difiere de la interna. Cuando se configura:

1. **Authorization URL:** usa `provider.redirectProxyUrl` como `redirect_uri` (en lugar de `provider.callbackUrl`)
2. **Token Exchange:** también usa `provider.redirectProxyUrl` como `redirect_uri`
3. Ambos son **idénticos** → Azure AD valida correctamente ✅

`provider.redirectProxyUrl` es computado por auth.js como:
```javascript
c.redirectProxyUrl = `${config.redirectProxyUrl}/callback/${c.id}`;
// → "https://core.nathium.com.ar/api/auth/callback/microsoft-entra-id"
```

`isOnRedirectProxy` se determina comparando el origin del `redirectProxyUrl` con el origin del request interno:
```javascript
isOnRedirectProxy = new URL(provider.redirectProxyUrl).origin === url.origin;
// "https://core.nathium.com.ar" === "https://0.0.0.0:3000" → FALSE
```

`isOnRedirectProxy = false` → se usa `redirectProxyUrl` como `redirect_uri` en ambos pasos ✅

### Cambio en `src/auth.ts`

```typescript
// ANTES (solo arreglaba la authorization URL, no el token exchange):
const dynamicProviders = [
  MicrosoftEntraID({
    clientId: process.env.AUTH_MICROSOFT_ENTRA_ID_ID,
    clientSecret: process.env.AUTH_MICROSOFT_ENTRA_ID_SECRET,
    issuer: `https://login.microsoftonline.com/${tenantId}/v2.0`,
    authorization: dynamicCallbackUrl ? { params: { redirect_uri: dynamicCallbackUrl } } : undefined,
  })
].filter(Boolean);

return {
  trustHost: true,
  providers: dynamicProviders,
  // ...
};
```

```typescript
// DESPUÉS (arregla ambos — authorization URL Y token exchange):
const dynamicProviders = [
  MicrosoftEntraID({
    clientId: process.env.AUTH_MICROSOFT_ENTRA_ID_ID,
    clientSecret: process.env.AUTH_MICROSOFT_ENTRA_ID_SECRET,
    issuer: `https://login.microsoftonline.com/${tenantId}/v2.0`,
    // Sin authorization.params.redirect_uri — redirectProxyUrl lo maneja
  })
].filter(Boolean);

const redirectProxyUrl = fullHost ? `${protocol}://${fullHost}/api/auth` : undefined;

return {
  trustHost: true,
  redirectProxyUrl,  // ← auth.js usa este valor para redirect_uri en authorization Y token exchange
  providers: dynamicProviders,
  // ...
};
```

### 2. `src/auth.ts` — `redirect` callback dentro del scope dinámico

Movido dentro de `NextAuth(async (req) => {...})` para tener acceso a `fullHost` y corregir redirects con `baseUrl=https://0.0.0.0:3000`.

```typescript
async redirect({ url, baseUrl }) {
  const hasWrongBase = baseUrl.includes("0.0.0.0") || baseUrl.includes("127.0.0.1");
  const effectiveBase = hasWrongBase && fullHost ? `${protocol}://${fullHost}` : baseUrl;
  if (url.startsWith("/")) return `${effectiveBase}${url}`;
  if (hasWrongBase && url.includes("http")) {
    const u = new URL(url);
    if (u.hostname === "0.0.0.0" || u.hostname === "127.0.0.1") {
      if (fullHost) return `${protocol}://${fullHost}${u.pathname}${u.search}`;
    }
  }
  // same-domain check...
  return effectiveBase;
}
```

## Restricciones de diseño importantes

- **No se puede usar `AUTH_URL`** — BizGuard maneja múltiples dominios públicos. Fijar una sola URL rompería todos los conectores excepto el primero.
- **No se puede usar `NEXTAUTH_URL`** — Misma razón.
- **No se puede configurar TLS en localhost:3000** — El servidor Next.js standalone recibe HTTP desde el proxy interno.
- La solución debe ser **completamente dinámica** por request, usando los headers `x-forwarded-host` y `x-forwarded-proto` para detectar el dominio público en cada request.
