# BizGuard v1.2.1 — Mejoras de Estabilidad, Métricas y UX
**Fecha:** 2026-03-21  
**Versión:** v1.2.1-Final  

## Resumen General

Sesión enfocada en resolver problemas críticos de estabilidad del dashboard, métricas de rendimiento y monitoreo de conectores. Se abordaron 5 fases de mejora continua (Fase 13–17).

---

## Fase 13: Sincronización de Métricas Robusta

### Problema
Las métricas de tráfico (requests, bytes) desaparecían del dashboard tras periodos de inactividad. Al recargar la página, a veces volvían.

### Causa
El flujo SSE (`EventSource`) se desconectaba silenciosamente por inactividad de red, y no había mecanismo de recuperación.

### Solución
- **Archivo:** `src/contexts/StatsContext.tsx`
  - Implementado sistema de **polling de respaldo** cada 5 segundos.
  - Si pasan más de 6 segundos sin datos del stream, se ejecuta un `fetch("/api/stats")` manual.
  - Si el stream permanece muerto 30+ segundos, se reconecta automáticamente el `EventSource`.
  - Se usan `useRef` para rastrear el timestamp del último dato recibido.

- **Archivo:** `src/app/api/stats/stream/route.ts`
  - Heartbeat reducido de 15s a **8s** para mantener conexiones TCP activas.

## Fase 14: Corrección de Consola y Formato de Logs

### Problema
En la consola de operaciones, el timestamp aparecía al final de la línea:
```
[INFO] [BIZGUARD-IN] GET /path -> core-serena
1:13:19 AM
```

### Solución
- **Archivo:** `src/components/GlobalConsole.tsx`
  - Reestructurado el layout para anteponer el timestamp:
    ```
    1:13:19 AM [INFO] [BIZGUARD-IN] GET /path -> core-serena
    ```
  - Timestamp con ancho fijo (`shrink-0 whitespace-nowrap`) para alineación visual.

## Fase 15: Métricas de CPU para Windows

### Problema
La métrica "Load" (CPU) siempre mostraba 0.0% en Windows.

### Causa
Se usaba `os.loadavg()` que retorna `[0, 0, 0]` en Windows (solo soportado en Linux/macOS).

### Solución
- **Archivo:** `src/lib/proxy-manager.ts`
  - Implementado cálculo basado en diferenciales de `os.cpus()`.
  - Se guardan los tiempos de CPU del ciclo anterior (`prevCpus`).
  - Se calcula: `cpuLoad = 100 * (1 - idleDiff / totalDiff)`.
  - Funciona tanto en Windows como en Linux.

## Fase 16: Fix Singleton ProxyManager en Producción

### Problema
El Origin Ping y métricas funcionaban en `npm run dev` pero NO en `dist-bizguard` (standalone build con `node server.js`).

### Causa
**Bug crítico** en la instanciación del singleton:
```typescript
// ANTES (bug)
export const proxyManager = global.proxyManager || new ProxyManager();
if (process.env.NODE_ENV !== "production") global.proxyManager = proxyManager;
```
En producción, `global.proxyManager` nunca se asignaba. Next.js standalone compila cada ruta API en un bundle separado, creando **múltiples instancias aisladas** de ProxyManager. La instancia que hacía los pings no era la que respondía a `/api/stats`.

### Solución
```typescript
// DESPUÉS (fix)
export const proxyManager = global.proxyManager || new ProxyManager();
global.proxyManager = proxyManager; // SIEMPRE compartir instancia
```

## Fase 17: Origin Ping — Detección Robusta y Z-Index Modal

### Problema 1: Ping siempre Offline en producción
Los servidores target respondían con **HTTP 500** al `HEAD /` (comportamiento normal de IIS/.NET). La condición `(statusCode < 500)` los marcaba como offline.

### Solución
```typescript
// ANTES: consideraba 500 como offline
const isOnline = (res.statusCode || 0) < 500;

// DESPUÉS: cualquier respuesta = servidor vivo
const isOnline = (res.statusCode || 0) > 0;
```
Lógica: si el servidor responde (aunque sea con error), está online. Offline = sin conexión.

### Problema 2: Modal editor tapado por la consola
El popup de edición de conectores tenía `z-[100]` y quedaba detrás de la consola.

### Solución
- **Archivo:** `src/components/ConnectorRow.tsx`
  - Z-index del modal subido a `z-[9999]`.

### Mejoras adicionales en ping
- **Auto-detección de protocolo**: Si la URL no tiene `http://` o `https://`, prueba HTTPS (443) y HTTP (80) automáticamente.
- **`rejectUnauthorized: false`** en el health-check de ping (solo ping, no tráfico real).
- **Logs de debug** detallados: `[PING-DEBUG]` y `[PING-RESULT]` con status code, latencia y errores específicos (`ECONNREFUSED`, `TIMEOUT`, etc.).
